# ADR 0002：文档治理与上下文记录机制

## 状态
Accepted

## 日期
2026-03-30

## 背景
- 项目已有较多模块与接口，知识分散在代码与口头沟通中。
- 仅靠临时沟通难以长期维护 AI 协作与多人并行开发的一致性。
- 需要一套轻量、可持续更新的文档机制。

## 备选方案
1. 不做统一文档，只靠 README 与代码注释。
2. 一次性编写全量大文档。
3. 建立固定骨架并按迭代增量维护。

## 决策
选择方案 3：固定骨架 + 增量维护。

骨架包含：
- `AGENTS.md`（协作约束）
- `PLANS.md`（任务上下文与决策过程）
- `docs/architecture.md`（架构与启动上下文）
- `docs/configuration.md`（配置治理）
- `docs/api/*.md`（接口文档）
- `docs/decisions/*.md`（ADR）
- `docs/changelog.md`（行为变更留痕）

## 影响
- 正向影响：
  - 关键决策可追溯，降低“信息只在人脑里”的风险。
  - 代码、配置、接口变更有统一同步机制。
  - 更适配 AI 协作场景（上下文连续性更好）。
- 负向影响：
  - 增加一定文档维护成本。
- 缓解措施：
  - 用“最小必要更新”原则，避免文档过度膨胀。
  - 将文档更新纳入 PR 检查项。

## 回滚与迁移
- 触发条件：文档维护成本显著高于收益，且团队无法执行。
- 回滚步骤：仅保留 `AGENTS.md + docs/api + changelog` 最小集合。
- 兼容性影响：无运行时影响，仅影响协作流程。
